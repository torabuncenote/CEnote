# Firebase セキュリティルール適用ガイド

CEnote が使う Firebase Realtime Database / Storage のセキュリティルール定義です。
これまでの推奨ルールは「ログイン済みなら全パス読み書き自由」でしたが、掲示板・
通知・バックアップなど機能が増えたのに合わせて、パスごとに必要最小限の権限へ
段階的に絞ったものです。

**このファイルはアプリからは自動配布されません。管理者が手動でFirebaseコン
ソールに貼り付けて適用する必要があります。**

## 適用手順

### Realtime Database

1. [Firebase Console](https://console.firebase.google.com/) を開き、対象プロジェクトを選択
2. 左メニュー「Realtime Database」→「ルール」タブ
3. `database.rules.json`（このリポジトリ直下）の中身を全選択してコピー
4. ルールエディタの内容を全て置き換えて貼り付け
5. 「公開」をクリック

### Storage

1. 左メニュー「Storage」→「ルール」タブ
2. `storage.rules`（このリポジトリ直下）の中身をコピー
3. ルールエディタに貼り付けて「公開」

## 適用後の動作確認チェックリスト

一般ユーザー（非管理者）アカウントでログインして、以下がすべて動くことを確認してください。

- [ ] 担当表・チェックリスト・業務内容カードの入力が保存される（リロードして反映を確認）
- [ ] 申し送りメモの投稿・画像添付ができる
- [ ] 掲示板への投稿・返信・画像添付ができる
- [ ] 掲示板の投稿を開くと既読（👀）が付く
- [ ] 承認待ち画像を投稿した際、管理者に通知が届く（管理者アカウントで確認）
- [ ] `@名前` メンションで本人に通知が届く
- [ ] ログが記録される（管理者の「ログ」タブで確認）
- [ ] 担当表タブの「タスク」サブタブでタスクの作成・ステータス変更ができる
- [ ] 自分が作成したタスクを削除できる（他人が作成したタスクは削除できない）

管理者アカウントでは追加で:

- [ ] スタッフの権限管理（`/userPerms` 読み書き）ができる
- [ ] Firebaseスナップショットの保存・一覧表示ができる
- [ ] 承認待ちメディアの承認・却下ができる

## パスごとの権限方針

| パス | 読み取り | 書き込み | 理由 |
|---|---|---|---|
| `/data`, `/data/pages/{ds}` | ログイン済み全員 | ログイン済み全員 | 連絡表本体。部署内共有データのため従来通り全員アクセス可 |
| `/board`, `/board/{id}` | ログイン済み全員 | 投稿者本人 or 管理者 | 他人の投稿を書き換え・削除させない |
| `/board/{id}/reads/{uid}` | ログイン済み全員 | 本人のuidのみ | 既読は自分の分しか書けない（なりすまし既読防止） |
| `/board/{id}/replies/{rid}` | ログイン済み全員 | 返信者本人 or 管理者 | 投稿と同様 |
| `/tasks`, `/tasks/{taskId}` | ログイン済み全員 | 作成・更新はログイン済み全員 / 削除は作成者本人 or 管理者 / `createdBy.uid` は作成後変更不可 | 担当者の名前配列とuidの照合はRTDBルールでは不可能なためアプリ側で制御し、ルールでは削除と作成者情報の改ざん防止を強制する（作成者を自分に書き換えてから削除する抜け道を塞ぐ） |
| `/backups`, `/backup_meta` | ログイン済み全員 | ログイン済み全員 | 中身は `/data` のスナップショットで機密性は同等。`cleanOldSnapshots()` が管理者以外のセッションからも定期実行されるため、admin限定にすると一般ユーザー使用中は自動整理が効かなくなる。既に `/data` 自体が全員読み取り可のため、実質的な追加露出はほぼ無い |
| `/recent_backup` | 管理者のみ | ログイン済み全員 | アプリからは書き込みのみで読み取りは行わない。念のため読み取りは絞る |
| `/logs` | ログイン済み全員 | 追記のみ（既存ログの上書き不可、削除は管理者のみ） | 改ざん防止。自動削除機能のための削除権限は管理者に限定 |
| `/admins` | ログイン済み全員 | 管理者のみ | 誰が管理者かは全員閲覧可（UI表示のため）だが、権限昇格は管理者のみ許可 |
| `/users`, `/users/{uid}` | ログイン済み全員 | 本人 or 管理者 | プロフィール（displayName等）は共有情報、書き込みは本人か管理者に限定 |
| `/userPerms` | 管理者のみ（全体） / 本人のみ（自分の分） | 管理者のみ | 権限情報は機微。本人は自分の権限だけ確認できる |
| `/notifications/{uid}` | 本人のみ | 本人以外も新規作成可（既読フラグ更新は本人のみ） | 通知は「他人に送る」操作が必須のため書き込みを開き、既読化・削除は本人限定 |
| `/fcmTokens/{uid}` | ログイン済み全員 | 本人のみ | プッシュ通知トークン。読み取りは通知送信ロジックのため全員可、書き込みは本人のみ |

### 既知の注意点

- **最初の管理者アカウントは Firebase コンソールから手動で `/admins/{uid}` に `true` を書き込んで作成してください。**（ルール上、管理者しか `/admins` に書き込めないため、ゼロ人目の管理者はアプリ経由では作れません）
- **`/tasks` パスはパス別ルールを最初に適用した時点では存在しませんでした。** パス別ルールは未定義パスをデフォルト拒否するため、タスク管理機能を使うには本ファイル更新後の `database.rules.json` を Firebase Console で再適用（上記「適用手順」を再実施）してください。再適用するまでタスクの作成・編集はローカルキャッシュにしか反映されず、他端末には同期されません。
- Storage 側はパス別の絞り込みを行っていません（`storage.rules` 内のコメント参照）。機微な情報（患者情報等）を画像で添付しないよう、引き続き PHI検知機能・運用ルールでカバーしてください。

## ロールバック

問題が起きた場合は、以前の簡易ルールに戻せます。

**Realtime Database（旧ルール）:**
```json
{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null"
  }
}
```

**Storage（変更なし・現行のまま）:**
```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /{allPaths=**} {
      allow read, write: if request.auth != null;
    }
  }
}
```
